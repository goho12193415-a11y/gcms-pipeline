## ★ 后续优化依据(现状 / 已证伪的路 / 仅剩的真杠杆)

### 当前性能(eval/score.py 裁定,是"与厂商/人工的一致度",非绝对准确率)

### 已穷尽 / 已证伪的提升路(别再重复试)
1. **谱图增强(多扫描平均-背景扣除)**:0/6,削特征离子,永久关。
2. **解卷积/成分感知谱**:对26个未命中 0/26 进top-30,无用。
3. **显示更多候选(top-10)**:答案根本不在mainlib深层池(0/30),无用。
4. **RI 重排(对NIST候选池重排序)**:A类正确同系物 0/11 在top-50——池里没有,重排无解。
5. **CNN预测RI主导注入**(`RI_INJECT_ALKANES`):净增0;预测准的峰谱图不像烷烃(门控不触发),谱图像烷烃的峰预测RI又分不开异构体。默认关。
6. **预测RI补noRI**(`RI_QC_USE_PREDICTED`):预测库与文献库99%重叠(仅+109),补不了noRI。默认开但实测≈0增益,留作接口。
7. **在线NIST WebBook补RI**:连通可解析,但noRI化合物(衍生物/支链烷烃/冷僻物)在WebBook同样没有(实测~2/20)。
8. **重解析NIST ri.dat扩库**:旧库本已较全;并集仅+357 WAX / +6357 DB-5。noRI是数据真没有,非解析不全。
9. **同进程双库(mainlib+replib)**:NIST DLL单库全局状态会互相污染(已证实),必须独立子进程。

### 剩余分歧的根因(软件层无解,占未命中大头)
- **A类支链烷烃(~21个)**:EI谱近全等 + DB-WAX无文献RI + 正确同系物不在NIST池 → 物理极限。
- **E/Z异构体**:四极杆物理极限。
- **超低信号长链醛/烯**:信噪比不足。
- **部分人工/厂商标注本身就是低置信猜测或自相矛盾**(如C13支链烷烃标在RI2421)。

### 仅剩的真·提升杠杆(都需实验配合,非纯软件)
1. **打标准品**:对关键化合物在你自己柱子上实测谱图+RI → 唯一能确证、且让RI_Check真正可信的办法。
2. **同序列标品**:RI标品与样品同批次跑 → RI_Check去噪、更多OK。
3. **更广的结构→RI预测模型**(比内置CNN覆盖更广的)→ 才可能补noRI;现有CNN无效。
4. **升级NIST库**(2020/2023,自带更多RI字段)→ 多覆盖一些。

### A类支链烷烃 = EI物理极限 (RI重排无解, 已证伪)
**门控实验** `diag/a_rank.py`: 11个A类支链烷烃, **正确化合物 0/11 出现在NIST top-50**。
**结论**: RI重排只能对已在候选池里的化合物重排序; 正确同系物根本不在池里(NIST只返回错碳数的同系物如Tetradecane-2,6,10-trimethyl, 或错取代位的Dodecane-2,6,11-trimethyl)。这正是上次 `ri_reranker.py` 失败的真因 — 不是RI数据少, 而是池里没有目标。
**RI主导注入已实现并测试** (`diag/a_ri_inject.py` + `nist_engine._inject_ri_alkanes`, config `RI_INJECT_ALKANES`): 用CNN预测RI(`ri_nist_full.json` name_to_ri)对谱图判定为烷烃的峰注入RI最近的支链烷烃候选。
**结果: 净增0, 默认关闭。** 两个互补的死结:
- 预测RI精准的峰(如2,6,10-Trimethylpentadecane |ΔRI|=4, 本应rank1注入), 其apex谱图根本不像烷烃(NIST返回醇类), 烷烃门控无法触发;
- 谱图确实像烷烃的峰(2,6,10-Trimethyldodecane), 正确同系物被预测RI更近的竞争者淹没(异构体ΔRI<预测误差)。
此外多个A类人工标注本身自相矛盾(同名出现在物理上不可能的RI, 如C13支链烷烃标在RI2421)——属人工Chromeleon猜测的错误, 任何算法都无法匹配。
**结论**: A类是 EI物理极限 + 缺文献RI + 人工标注误差 三重叠加, 软件层无解。代码与flag保留, 待更准的RI模型。

## 2026-06-29 续: 跨仪器(.qgd) / RI库扩充 / 审核分级 / 自标定RI_Check

### 岛津 .qgd 解码 (`qgd_reader.py`) — 维护要点
- `.qgd` = OLE2 复合文档(`olefile` 读)。MS数据在 `GCMS Raw Data/` 下:`Retention Time`(int32 ms/扫描)、`Spectrum Index`(int32 字节偏移/扫描)、`MS Raw Data`(谱)、`TIC Data`(int32,每隔一个取)。
- 每扫描:24字节头 + (uint16 m/z×20, uintN 强度) 点;强度**自适应**:低强度4字节(2字节强度),高强度5字节(3字节强度,头32字节)。
- **关键技巧**:用 `TIC Data` 作每扫描校验和——两种解法都试,强度和==TIC 的那个为准 → 5400/5400 全过,零静默错误。
- **噪声底**:岛津原始谱每个 m/z 有恒定~500计数噪声底(Thermo mzML已预阈值,无此问题),不去掉会让NIST匹配成巨型分子。用**每扫描中位数×3**阈值滤除(`_NOISE_MULT`)。m/z = col0/20;RT = ms/60000。

### 自标定 RI_Check (step9_export) — 现行逻辑
- 不再用写死的Thermo变换。**每个样品**用自身高置信命中(RMF≥850且有文献RI)做 **Theil-Sen 鲁棒拟合** `litRI=a·measRI+b`(抗离群、不过拟合;迭代裁剪会过拟合致误判,已弃)。
- 每峰top-1:`OK`/`suspect`(对文献RI,强)→ 无文献则 `OK-pred`/`suspect-pred`(对CNN预测RI,弱,松阈值)→ 都没有则 `noRI`;样品锚点不足则 `noCal`。

### 2026-07-02 独立准确率验证 (eval/bench_massbank.py)
- **动机**:用户不想跑标准品,问能否在线拿到真实验证。做法:把 **MassBank(北美/EU)独立采集的 EI 库谱**(13,534 条,独立于 NIST)丢进本机 NIST 搜索,按 **CAS/InChIKey 结构**判定命中(不是名字),消除命名假未命中。
- **踩坑记录**(为什么名字比对不行):MassBank 名字多为大写 IUPAC/异构写法(如 `4-METHYL-2-PENTANONE`)、NIST 用反转命名(`2-Pentanone, 4-methyl-`)、还有大量常用名(vanillin↔4-羟基-3-甲氧基苯甲醛)。纯名字比对 top-1 仅 **11%**→加命名反转归一 18%→加 NIST `synonyms` 39%→**加 PubChem 结构(CAS)对接 49%**。即前面的"低"绝大部分是**测量假象**。MassBank↔NIST 无共享结构ID(MassBank 有 InChIKey 无 CAS;NIST DLL 给 CAS 无 InChIKey),用 **PubChem REST**(InChIKey→CAS/synonyms)搭桥。
- **结果**(NIST14 mainlib,未衍生化 + MW≤200 ≈ 类挥发物,n=300,结构核对):
  - 原始:**top-1≈60% / top-5≈70%**
  - 分解:**可检索率 71%**(正确物在 top-30) × **给定可检索 top-1≈89% / top-5≈100%**
  - **结论:排序引擎接近极限(库里有的东西 89% 排第一、~100% 进前5);整体天花板 = 库覆盖 + EI 同分异构歧义,不是算法。**
- **边界/诚实**:①库谱比真实色谱峰干净 → 对真实样品是**偏乐观上限**;②MS-only,未用 RI,而真实管道的 MS+RI 双证据在异构体上更强 → 真实场景可更高;③MassBank EI 偏代谢组衍生物(已滤),仍非用户样品分布。
- **意义**:这是个**不循环、可复现、且对项目有利**的独立验证,反过来说明之前和人工 top-1 仅 41% 一致主要是**人工端**(疲劳/只选 top1)拉低,非引擎问题。

### replib 复本库合并 (top-8) + 关键架构坑

**NIST DLL 单库全局状态污染 (已证实)**: `pyms_nist_search` 封装的 DLL 只有一个全局"当前库"。同一进程内创建第二个 Engine(replib) 会**劫持** mainlib —— 之后 mainlib.search 实际返回 replib 结果 (实测: mainlib 对某谱图本应返回 1-Octen-3-one RMF895, 第二引擎创建后变成 replib 的 Octadecane RMF654)。所以同进程双库**不可行**, 会让覆盖率从 23 掉到 22。
**可靠方案**: `replib_pass.py` 在**独立子进程**中只搜 replib, dump 候选; `pipeline.merge_replib_candidates` 离线合并。
**合并策略 (reserve, 防位移)**: 保留 mainlib 的前 `keep=top_n-reserve` 名不动 (top-5 的 23 个零损失), 仅在 rank 6-8 注入 replib 独有候选; 且只对 mainlib 不自信的峰(top-1 RMF<850)注入, 并设 RMF≥750 下限去噪。净增 3 个 (2,3-Octanedione / (Z)-7-Hexadecenal / 二异丁酸酯), top-8 = 26/49。
**配置**: `config.NIST_USE_REPLIB / TOP_N_CANDIDATES=8 / NIST_REPLIB_RESERVE=3 / NIST_REPLIB_FALLBACK_RMF=850 / NIST_REPLIB_MIN_RMF=750`。
**展开复核窗口本身无用**: mainlib top-8 仍是 23 (未命中答案根本不在 mainlib 深层排名里, 见 `diag/miss_rank.py` 的 0/26)，只有 replib 才能填进新增的 rank 6-8。Excel 新增 `Source` 列标注 replib 来源。
